üîë TYPES OF INHERITANCE (THE DIFFERENCE)

‚úÖ Public Inheritance (What You're Using)

class ScavTrap : public ClapTrap

What It Means:

  • Public members of ClapTrap ‚Üí Public in ScavTrap
  • Protected members of ClapTrap ‚Üí Protected in ScavTrap
  • ScavTrap IS A ClapTrap (inheritance relationship is visible)

Use When: You want the child to be a specialized version of the parent (like you're doing!)

‚ö†Ô∏è Protected Inheritance (Rare)

class ScavTrap : protected ClapTrap

What It Means:

  • Public members of ClapTrap ‚Üí Protected in ScavTrap
  • Protected members of ClapTrap ‚Üí Protected in ScavTrap
  • The "IS A" relationship becomes hidden from outside

Use When: You want to inherit implementation but hide the relationship from users.

üîí Private Inheritance (Also Rare)

class ScavTrap : private ClapTrap

What It Means:

  • Public members of ClapTrap ‚Üí Private in ScavTrap
  • Protected members of ClapTrap ‚Üí Private in ScavTrap
  • Complete hiding - used for implementation inheritance only

Use When: You want to reuse code but completely hide that ScavTrap is based on ClapTrap.

üí¨ WHAT TO TELL THE EVALUATOR

"Where Does Inheritance Happen?"

"Inheritance happens in ex01 and ex02:

  • In ScavTrap.hpp line 18: class ScavTrap : public ClapTrap
  • In FragTrap.hpp line 18: class FragTrap : public ClapTrap

Both classes use public inheritance, which means ScavTrap and FragTrap are types of ClapTrap - they inherit all its features and can add their own."

"What's the Difference Between Public/Protected/Private Inheritance?"

"Public inheritance (what I used) maintains the 'IS A' relationship - ScavTrap is a ClapTrap. The parent's public interface stays public.

Protected and private inheritance are for hiding the relationship - they make parent members more restrictive. You'd use them when you want to reuse implementation without exposing that the classes are related.

For this project, public inheritance is correct because ScavTrap and FragTrap truly are specialized versions of ClapTrap."

"Why Did You Change Attributes from Private to Protected?"

"In ex00, attributes are private because only ClapTrap needs to access them. In ex01 and ex02, I changed them to protected so that child classes (ScavTrap and FragTrap) can access them directly. This is more efficient than using getters and setters, and it's the proper way to share data with derived classes while still keeping it hidden from external code."

"Explain the Constructor/Destructor Chaining"

"When you create a ScavTrap:

  1. ClapTrap constructor runs first (builds the base)
  2. ScavTrap constructor runs (adds the upgrades)

When you delete a ScavTrap:

  1. ScavTrap destructor runs first
  2. ClapTrap destructor runs (cleans up the base)

This is visible in the output messages and ensures proper construction and cleanup order!"
