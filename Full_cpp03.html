CPP03 Master Sheet (ex00–ex02)

Goal: understand single inheritance deeply (not memorize). Use the left menu to jump around.

Why C++ after C?

In C
  • Data lives in struct, behavior is separate functions
  • You must manually enforce “rules” (who can change what)
  • Copying and resource handling is manual discipline
In C++
  • Data + behavior packaged into a class
  • Access control: private/protected/public
  • Constructors/destructors enforce valid state automatically
Key idea (scientific phrasing)

C++ adds a formal mechanism to maintain invariants: “this object must always be in a valid state.” CPP03 tests you on modeling these invariants across a hierarchy.

What is a Class (mental model)

Class = blueprint. It describes:

  • State: variables (attributes)
  • Behavior: functions (methods)
  • Rules: who can access what (encapsulation)
Blueprint (Class)         Instances (Objects)
-----------------         -------------------
class ClapTrap { ... }    ClapTrap a("A");
                          ClapTrap b("B");
Mentor rule: If you can answer “What does this class OWN?” you will design it correctly.

What is an Object

Object = one real instance that has:

  • Current state (values of HP/energy/name/etc.)
  • Allowed operations (attack, repair, take damage)
Object "a"
---------
_name        = "A"
_hitPoints   = 10
_energyPoints= 10
_attackDamage= 0

Operations:
- attack("B")
- takeDamage(3)
- beRepaired(2)

Why Constructors exist

Constructor = “birth”

Creates a valid object by setting required initial values.

Destructor = “death”

Final step when object ends. In CPP03, mainly prints messages + shows order.

ConceptMeaningWhy CPP03 cares
ClapTrap() Default birth Lets you create then assign later; common in C++ patterns
ClapTrap(name) Birth with identity Most exercises create named objects
~ClapTrap() Death Shows destruction order (esp. with inheritance)
Important: A constructor is not “repeat initialization”. It’s the only moment you can guarantee the object becomes valid from scratch.

Canonical Form (WHY not WHAT)

Orthodox Canonical Form exists because copying objects is a real operation in C++.

You must define how your object behaves when it is:

  • copied at creation (copy constructor)
  • copied after creation (copy assignment)
  • destroyed (destructor)
FunctionWhen it happensWhy it exists
T(const T&) New object is born from another Defines the “clone” operation
operator=(const T&) Existing object becomes like another Defines replacement without re-birth
~T() Object ends Ensures cleanup + teaches order
Your correct insight: Self-assignment check prevents useless work and avoids corruption in complex classes.

Encapsulation (private / protected / public)

AccessWho can accessWhen to use
private Only the class itself Strongest protection for invariants
protected Class + derived classes Inheritance-friendly (CPP03 uses this often)
public Everyone API: what outside code is allowed to do
CPP03 reasoning:

ClapTrap attributes are often protected so children can configure initial stats (HP/Energy/Damage) without creating duplicate data.

Evaluation red flag: Making everything public to “make it easier”. This breaks encapsulation and signals weak OOP.

Inheritance (IS-A vs HAS-A)

IS-A (inheritance)

Use when the child is a specialized form of the parent.

ScavTrap IS A ClapTrap
FragTrap IS A ClapTrap
HAS-A (composition)

Use when a class contains another as a part.

Car HAS AN Engine
School HAS Students

ClapTrap
  ├─ attack(), takeDamage(), beRepaired()
  ├─ _name, _hitPoints, _energyPoints, _attackDamage
  ├─ default stats (10/10/0)
  |
  ├─ ScavTrap  (100/50/20) + guardGate()
  |
  └─ FragTrap  (100/100/30) + highFivesGuys()
Your mastered rule: Base stores the data; derived chooses initial values.

Why public inheritance matters

Two different “public/protected/private” ideas exist:

  • Member access: private/protected/public inside the class
  • Inheritance mode: class Child : public Parent
Inheritance modeMeaningCPP03 fit?
public Parent Child IS-A Parent (true type relationship) YES
protected Parent Outside can’t treat Child as Parent No
private Parent Child “uses” Parent internally (not IS-A publicly) No
Mentor sentence (use in evaluation): Public inheritance preserves the IS-A relationship; ScavTrap must be usable as a ClapTrap.

Constructor chaining (visual)

Rule: Base constructor runs first. Derived constructor body runs second.

Create ScavTrap("Bob")

1) ClapTrap("Bob")   <-- base part is built first
2) ScavTrap body     <-- then child customizes stats/messages

Why : ClapTrap(name)?

It’s the only way to choose which base constructor is called. If you wait inside { }, the base already got constructed. 

ScavTrap::ScavTrap(const std::string &name) : ClapTrap(name)
{
    // body runs after base is done
}
Syntax reminder: Never put spaces inside ::. It’s one operator (scope resolution).

Function overriding (mental rule)

Override in CPP03 means:

  • Same function name
  • Same signature (parameters + constness)
  • Child provides new behavior/messages
Mental lookup rule 
s.attack()

Look in ScavTrap:
- found attack()? YES -> use it
- NO -> fallback to ClapTrap::attack()
Limit (before CPP04)

Without virtual, dynamic dispatch isn’t the topic yet. CPP03 focuses on direct object calls and inheritance structure.

Common Confusions (YOUR questions) + clean answers

1) “Why is attack damage 0 in ex00?”

Because ex00 initializes damage to 0 by requirement. Damage becomes meaningful in ScavTrap/FragTrap.

2) “IS-A means object inside object?”

No. IS-A means “can be treated as”. A ScavTrap can be viewed as a ClapTrap type.

3) “Why do we write : ClapTrap(name)?”

Base must be constructed before derived body runs. The initializer list is the only place to choose the base constructor.

4) “Can I put spaces in ::?”

No. :: is a single operator; spacing inside it breaks syntax.

5) “Why do we need default constructor if we always pass a name?”

C++ allows create-then-assign patterns. A default constructor makes the type usable in more contexts.

6) “Why prefix members with underscore?”

Convention to mark member variables and avoid name collisions with parameters (readability + reduces shadowing).

7) “Why include ScavTrap files in ex02 if not used?”

Because later exercises include previous classes; evaluation can compile and test them indirectly.

8) “I understand while reading but might forget on a blank page.”

Normal. The fix is writing from prompts: “Who owns this function?” → write ClassName::, then copy the exact signature from the header.

When inheritance is used in real life

DomainBaseDerivedWhy it fits IS-A
UI frameworks Widget Button, TextBox All share draw/position; behavior differs
File handling File ImageFile, AudioFile All have name/size/open; specialized processing
Networking Connection TCPConnection, UDPConnection Same API; different protocol behavior
Error hierarchies Error FileError, NetworkError Shared message/code; specialized meaning
One-sentence scientific summary

Inheritance models specialization under a shared interface while preserving base invariants and reducing duplicated logic.

When NOT to use inheritance

Prefer composition (HAS-A) when…
  • You only want to reuse a small feature, not an identity
  • The relationship is “uses”, not “is”
  • You expect many combinations of features
Red flags for inheritance
  • Child doesn’t behave like parent (IS-A fails)
  • You override many functions just to disable them
  • You’re inheriting only to reuse code, not model reality
Practical modern rule: Use inheritance sparingly (often 1 level deep). CPP03 teaches the core mechanics.

Quick “blank page” writing drills

Drill A: Write an override from scratch
  1. Copy signature from header: void attack(const std::string &target);
  2. Choose owner class → write void ClassName::attack(...)
  3. Paste the same rule checks as base (HP/energy)
  4. Change only the message prefix (ClapTrap/ScavTrap/FragTrap)
Drill B: Write a derived constructor skeleton 
Derived::Derived(const std::string &name) : Base(name)

    // message
    // set stats
}
